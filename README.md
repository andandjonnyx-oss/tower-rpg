\# Tower RPG



スマホ向けのノンフィールドRPGを Unity で開発中。

プレイヤーは塔を1歩ずつ進み、ランダムエンカウント・会話イベント・アイテム取得などを経て上階を目指す。



\---



\## Project Overview



このプロジェクトは、フィールド移動型ではなく、

\*\*「塔を進む」ことに特化したノンフィールドRPG\*\* である。



\### 基本ループ

1\. 塔シーンで1歩進む

2\. イベント抽選

&#x20;  - ランダムエンカウント

&#x20;  - 会話イベント

&#x20;  - アイテム取得

&#x20;  - 何も起きない

3\. 必要に応じて戦闘シーンやUIを表示

4\. 処理後、塔シーン進行に戻る



\---



\## Current Development Scope



現在はゲーム全体の骨組みを優先して実装している。



\### 現在の主な実装対象

\- 塔の進行

\- ランダムエンカウント

\- 戦闘シーンへの遷移

\- 会話イベントの管理

\- アイテム取得とアイテムBOXの基礎

\- 装備データの保持

\- シーン切替後も保持される進行情報の管理



\### 現時点の方針

\- まずは 1～20階程度までの骨組みを作る

\- 後からデータや演出を追加する

\- 会話内容・敵データ・アイテムデータは後から増やせる構成を目指す



\---



\## Core Scenes



\### Tower Scene

メイン進行シーン。

プレイヤーは塔を step 単位で進行する。



\#### 主な役割

\- 現在の floor / step の管理

\- ボタン入力による進行

\- イベント抽選

\- アイテムBOX表示

\- 会話イベント表示

\- 戦闘発生時の Battle Scene への遷移



\---



\### Battle Scene

戦闘専用シーン。



\#### 主な役割

\- 戦闘中の敵表示

\- プレイヤーと敵の戦闘処理

\- 勝敗判定

\- 戦闘終了後の塔シーン復帰



\---



\## Main Systems



\### Encounter System

ランダムエンカウント管理システム。



\#### Responsibilities

\- 塔進行直後に戦闘発生判定を行う

\- 現在の floor / step に応じて出現可能な敵を選ぶ

\- 戦闘対象データを保持し Battle Scene に渡す



\#### Related Data

\- MonsterDatabase

\- Monster



\---



\### Talk Event System

会話イベント管理システム。



\#### Responsibilities

\- floor / step 条件に応じた会話イベントの検索

\- 該当イベントの表示

\- データベース方式でイベントを追加可能にする



\#### Related Data

\- TalkEvent

\- TalkEventDatabase

\- EventTrigger / Runner 系の制御クラス



\---



\### Item System

アイテム取得・保持・使用・装備管理システム。



\#### Current Scope

\- 塔進行中にアイテム取得イベントが発生する

\- ポップアップで「拾う / 無視」を選択

\- 拾ったアイテムはアイテムBOXに格納

\- アイテムBOXで選択中アイテムの詳細を下部に表示

\- アイテム種別に応じてボタン文言を変更

&#x20; - アイテム: 使用 / 捨てる

&#x20; - 武器: 装備 / 捨てる

&#x20; - 魔法: 使用不可または別処理

\- 装備状態はシーン切替後も保持

\- 装備中アイテムはUI上で識別可能にする



\#### Planned Behavior

\- 所持上限がある場合、満杯時は「これ以上持てない」を表示

\- 必要に応じて既存アイテムを捨てる選択へ遷移

\## Important Data Design



\### Monster

敵1体分のデータを持つ ScriptableObject。



想定内容:

\- ID

\- 名前

\- 画像

\- 出現階層・step範囲

\- 出現重み

\- 各種ステータス



\---



\### MonsterDatabase

敵データ一覧を保持し、

現在の floor / step に応じて出現候補を返す。



\---



\### TalkEvent

会話イベント1件分のデータ。



想定内容:

\- ID

\- 発生階層

\- 発生step

\- 表示内容

\- 一度きりかどうか

\- その他条件



\---



\### TalkEventDatabase

TalkEvent の一覧を保持し、

条件に一致するイベントを取得する。



\---



\### Item Data

アイテム・武器・魔法などの基礎データ。

ScriptableObject または同等のデータ構造で管理する想定。



想定内容:

\- ID

\- 名前

\- 種別

\- 説明文

\- アイコン

\- ソート順

\- 使用可否

\- 装備可否



\---



\## State Persistence



このプロジェクトでは、シーン切替後も保持したい情報がある。



\### Persistent State Examples

\- 現在の floor / step

\- 戦闘前の進行位置

\- 現在装備中の武器

\- 所持アイテム

\- 一度発生したイベントの記録

\- 次のシーンへ渡す敵情報



永続情報は、単一の管理クラスまたはセーブ対象データとして保持する想定。



\---



\## Coding Intent



このプロジェクトでは、\*\*あとからデータ追加しやすい構成\*\* を重視している。



\### Design Priorities

1\. 骨組みを先に作る

2\. データ追加で拡張できるようにする

3\. シーンごとの責務を明確にする

4\. ScriptableObject を活用して管理しやすくする

5\. AIや外部開発者が読んでも構造を理解しやすくする



\---



\## Notes for AI Assistants



このリポジトリを読むAIへ。



\### Important Rules

\- 既存コメントは削除しない

\- 不要なリファクタリングをしない

\- 命名を勝手に変更しない

\- 既存処理の責務分離を尊重する

\- 可能な限り「追記」や「差分修正」を優先する

\- Unity 向けの一般的な構成を守る

\- Scene / ScriptableObject / UI の関係を意識して提案する

\- 単独ファイルだけでなく、関連クラスとの接続も考慮する



\### Preferred Support Style

\- まず既存構造を説明する

\- 次に変更箇所を限定して提案する

\- 必要なら新規スクリプトを追加する

\- 破壊的な全書き換えは避ける
